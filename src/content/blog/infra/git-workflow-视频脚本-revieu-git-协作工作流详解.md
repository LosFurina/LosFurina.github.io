| **0:48** | **[CI/CD 流程动画]**<br><br>  <br><br>1. 代码块并入 `dev` -> 屏幕显示 "Deploying to Test Env"。<br><br>  <br><br>2. 代码块并入 `main` -> 屏幕显示 "Deploying to Production"。                 | 为什么要这么分呢？这就好比是一个安全网。<br><br>如果我们直接合并新代码到 main 分支，那么会直接触发 main 分支的 CICD 流程，新代码总是不可避免的会存在一些隐藏 bug，而 main 分支的内容是已经发布在服务器正在运行的，一旦触发 CICD 自动部署，有未知 bug 的代码一定会影响到正在运行的服务。此外 main 分支应该最大程度保持整洁，因为涉及到多人合作开发，而 main 分支就是合作的 baseline，如果 main 分支随意合并随意提交，那么 git conflict 将会不可避免的多次出现，经常用 git 合作的小伙伴一定不想看到莫名的 conflict 吧<br><br>所以，大家日常写的新代码，都是先合并到 `dev` 分支。这时候，CI/CD 会自动把代码部署到**测试环境**。<br><br>  <br><br>你们可以在这里尽情地测试、折腾。<br><br>  <br><br>只有当团队确信 `dev` 上的代码已经足够稳定了，我们才会把它合并到 `main`，发布给真实用户。 |

## 第二部分：命名与提交规范 (Naming & Commit Standards)

| **时间**   | **画面内容 (Visual)**                                                                                                                                                                                                                           | **口播台词 (Spoken Script)**                                                                                                                                                                                                                                                                                                         |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1:45** | **[分支命名演示]**<br><br>  <br><br>屏幕显示公式：<br><br>  <br><br>`type / issue-id - description`<br><br>  <br><br>此时弹出一个 GitHub Issue 截图，ID 为 #42。                                                                                                    | 接下来讲讲规矩。为了让代码可追溯，分支命名必须遵循一个公式：<br><br>  <br><br>**类型，斜杠，Issue ID，再加简短描述**。<br><br>  <br><br>比如，你领到了一个 ID 为 42 的登录页需求，那你的分支名就必须是：`feat/42-user-login`。<br><br>  <br><br>加上这个 ID 非常关键，这样 GitHub 就会自动把你的代码和需求单关联起来，不管是做 Code Review 还是以后查问题，都能一键跳转。                                                                                 |
| **2:15** | **[Commit Message 对比]**<br><br>  <br><br>左边（错误示范）：<br><br>  <br><br>❌ `update`<br><br>  <br><br>❌ `fix bug`<br><br>  <br><br>右边（正确示范）：<br><br>  <br><br>✅ `feat(auth): add google oauth`<br><br>  <br><br>✅ `fix(ui): fix button alignment` | 同样严格的还有 **Commit Message**。<br><br>  <br><br>请大家千万不要再写什么 "update" 或者 "fix bug" 这种毫无意义的废话了。<br><br>  <br><br>我们强制遵循 **Conventional Commits** 规范。<br><br>  <br><br>格式是：`type(scope): subject`。<br><br>  <br><br>比如 `feat(auth): add google oauth`。<br><br>  <br><br>这不仅是为了好看，更是因为我们的发布系统会根据这些前缀，**自动生成 Changelog**。你写得规范，文档就自动生成了。 |

### 💡 给录制者的建议 (Tips for the Speaker)

1. **语速**：不要太快，特别是在念代码命令（如 `git checkout -b`）的时候，稍微停顿一下，给观众消化的时间。
    
2. **重音**：脚本中加粗的字（如 **永久存在**、**Issue ID**、**dev**）请重读，以起到强调作用。
    
3. **语气**：在批评错误示范（如 `fix bug`）时，可以用一种轻松调侃的语气，不要太严厉；在讲到 `main` 分支保护时，要严肃认真。
